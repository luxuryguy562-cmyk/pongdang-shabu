# 퐁당샤브 관리 시스템 — 헌법 (CLAUDE.md)

> **이 문서가 최상위 규칙이다.** .clauderules는 존재하지 않는다.
> 에이전트 지침(agents/*.md)과 충돌하면 이 문서가 이긴다.

---

## 제1조. 절대 규칙

### 1-1. 승인 게이트
- 계획서를 사장님에게 보여주고 **"OK" 받기 전에는 코드를 한 줄도 수정하지 않는다.**
- 사장님이 수정 요청하면 계획서를 고쳐서 다시 제출한다.
- 승인은 반드시 대화창에서 명시적으로 받는다. 암묵적 동의는 없다.

### 1-2. 브랜치 운영
- `main` 브랜치에 직접 push 금지.
- 작업은 `claude/` 또는 `feature/기능명` 브랜치에서.
- **브랜치 push 완료 → main 자동 머지**
  (Cloudflare Pages 배포 브랜치가 main이라, 사장님이 앱에서 테스트하려면 main에 올라가야 함)
- 예외: 사장님이 "머지 보류" 또는 "브랜치에만" 명시하면 머지 중단.
- 계획서 승인 게이트(1-1, 4-3)는 그대로 — **계획 승인 → 코드 작성 → 브랜치 push → 자동 main 머지 → 사장님 앱에서 테스트**.

### 1-3. 백업 필수
- 코드 수정 전 현재 상태를 커밋(백업)한다.
- DB 스키마 변경은 실행 SQL + 롤백 SQL 을 계획서에 포함한다.

### 1-4. 단일 파일 SPA 유지
- `index.html` 하나에 HTML+CSS+JS 전부. 파일 분리하려면 별도 승인.

### 1-5. 기존 기능 보호
- 작동 중인 기능을 깨뜨리면 즉시 롤백한다.
- 기존 함수명, 변수명, DOM ID를 무단 변경하지 않는다.

## 제2조. 우선순위 (충돌 시)

1. **수익화** — 기능이 돈이 되는 방향
2. **안정성** — 오류 제로, 기존 기능 보호
3. **단순함** — 복잡한 구조 금지
4. **자동화** — 반복 작업은 자동화

## 제4조. 에이전트 호출 절차 (워크플로우)

### 4-1. 작업 규모 분류

| 규모 | 기준 | 경로 | 토큰 절약 |
|------|------|------|-----------|
| **소형** | 텍스트 수정, 색상 변경, 버그 1줄 수정 | planner → coder → deployer | reviewer/tester/designer 생략 |
| **중형** | 기존 기능 개선, UI 추가, 쿼리 변경 | context_reader → advisor → planner → designer(UI있으면) → reviewer → coder → tester → deployer | 전체 경로 |
| **대형** | 새 탭/모듈 추가, DB 스키마 변경 | context_reader → advisor → planner → designer → reviewer → **사장님 승인** → coder → tester → deployer | 전체 경로 + 승인 게이트 강화 |

**규모 판단 기준:**
- 변경 라인 10줄 이하 → 소형
- 변경 라인 10~100줄 또는 DB SELECT만 → 중형
- 변경 라인 100줄 이상 또는 DB 스키마 변경 → 대형
- 판단이 애매하면 한 단계 위로 (중형→대형)

### 4-2. 호출 순서도

```
사장님 요청
  │
  ├─ [소형] planner가 간단 계획 → 승인 → coder → deployer
  │
  └─ [중형/대형]
       │
       ▼
  ① context_reader ─── 현황 보고서 작성
       │
       ▼
  ② advisor ────────── 기술 추천 (더 좋은 방법 있으면 제안)
       │
       ▼
  ③ planner ────────── 계획서 작성 (advisor 추천 반영)
       │
       ├─ UI 변경 있으면 ──→ ④ designer ── UI 설계 추가
       │                          │
       ▼                          ▼
  ⑤ reviewer ───────── 체크 (반려 가능 → ③으로 복귀)
       │
       ▼
  ⛔ 계획서 제출 ────── 사장님 확인 (여기서 멈춤)
       │
       │ "OK" 승인
       ▼
  ⑥ coder ──────────── 코드 구현
       │
       ▼
  ⑦ tester ─────────── 검증 (실패 → ⑥로 반환)
       │
       ▼
  ⑧ deployer ──────── push & 배포 확인
       │
       ▼
  사장님 결과 확인 🎉
```

### 4-2a. 에이전트 사고 과정 출력 규칙
- 각 에이전트 역할 수행 시 반드시 **`📋 [에이전트명]`** 헤더를 출력한다.
- 헤더 아래에 해당 에이전트의 핵심 판단 내용을 3~5줄로 요약한다.
- 다음 에이전트로 넘어갈 때 `→ [다음 에이전트명]에게 전달` 을 명시한다.
- 사장님이 "상세" 또는 "자세히"라고 하면 줄 수 제한 없이 전체 출력한다.
- 사장님이 "생략" 또는 "빨리"라고 하면 최종 계획서만 출력한다.

### 4-3. 승인 게이트 규칙
- **소형**: 계획을 한 줄로 요약해서 보여주고 바로 진행 가능 (묵시적 동의 가능)
- **중형**: 계획서 제출 후 승인 대기
- **대형**: 계획서 + DB 변경 상세 + 리스크 분석까지 제출 후 승인 대기

## 제5조. 에이전트 간 핸드오프 (Context Hand-off)

### 5-1. 원칙
에이전트가 다음 에이전트에게 작업을 넘길 때, 반드시 **구조화된 인수인계 블록**을 포함한다.

### 5-2. 인수인계 블록 양식

```
---HANDOFF---
from: [현재 에이전트]
to: [다음 에이전트]
작업: [작업 제목]
핵심 결정사항:
  - [결정 1]
  - [결정 2]
변경 대상:
  - 함수: [함수명 목록]
  - DOM: [ID 목록]
  - DB: [테이블.컬럼 목록]
주의사항:
  - [놓치면 안 되는 것]
---END HANDOFF---
```

### 5-3. coder 의무
- coder는 인수인계 블록의 **변경 대상**을 하나씩 체크하며 구현한다.
- 구현 완료 후 각 항목 옆에 ✅ 표시한다.
- 인수인계 블록에 없는 항목은 구현하지 않는다.

### 5-4. 작업 로그 (세션 끊김 대비)
- 중형/대형 작업 시 `docs/work_log.md`에 현재 진행 상태를 기록한다.
- 기록 시점: ① 계획서 완성 ② 승인 ③ 구현 완료 ④ 배포 완료

```
## 작업 로그

### [날짜] [작업 제목]
- 상태: 계획서작성중 / 승인대기 / 구현중 / 검증중 / 배포완료
- 브랜치: feature/기능명
- 계획서 요약: (한 줄)
- 마지막 완료 단계: [에이전트명]
- 다음 단계: [에이전트명]
```

## 제7조. 코딩 규칙

1. HTML 테이블 — 표 데이터는 `<table>` 태그 필수 (CSS grid 금지)
2. 금액 표시 — 만원 단위, 숫자만 (`125` = 125만원)
3. 네이밍 — camelCase, 한글 주석 필수
4. Supabase — `sb.from('테이블')` 패턴, `.eq('store_id', currentStore.id)` 필수
5. 에러 — 기술 에러 노출 금지, `alert('저장 실패')` 수준
6. 새 함수 구분 — `// ─── 새 기능: 기능명 ───` 주석

## 제10조. 운영 원칙 (구 .clauderules 통합)

1. **데이터 무결성** — 이름/담당자/소속 변경 시 기존 데이터 유지. 오염 방어 로직 선행.
2. **관리 편의성** — 코드 수정 없이 데이터 변경 반영되는 구조.
3. **자가 치유** — 오류 발생 시 사용자에게 전달하지 말고 자체 해결 반복.
4. **선제적 제안** — 더 효율적이고 비용 적은 방법을 먼저 제안.
5. **법무/리스크** — 개인정보·결제·계약 데이터는 안전 저장. 완성 단계에서 자동 적용.
6. **재무 연결** — 모든 기능은 매출/비용/수익과 연결.
7. **세무 자동화** — 거래 자동 기록, 세무 신고 형태로 저장.
8. **인건비 최소화** — 입력→처리→기록→보고 자동 흐름.
9. **확장성** — 여러 매장, 여러 사용자 기준 설계.
10. **수익화** — 무료→유료 전환 구조, 구독/수수료 모델 항상 고려.
11. **의사결정 기준** — 단순 > 저비용 > 쉬운유지보수 > 자동화 > 수익 > 저리스크 > 안정성.
12. **로그/추적** — 변경사항 기록, 1분 내 원인 추적 가능, 재발 방지 로직 자동 생성.
13. **UX** — 직관적 사용, 입력 최소화, 자동완성/추천 활용.

## 제11조. 대규모 변경 안전 절차 (리팩터링·일괄 치환)

> 2026-04-17 Phase 1 인라인 핸들러 251개 제거 작업에서 검증된 절차.

### 11-1. 사전 스캔 (측정 먼저)
- `grep`으로 변경 대상 **총 개수** + **유니크 패턴 수** 측정
- 유니크 패턴 리스트 저장 (`sort -u`) → 카테고리화
  - A. 단순 `foo()` / `foo(args)`
  - B. 복합(다중 호출·JS 표현식)
  - C. 특수(전역 변수 참조·동적 인자)
- 복합·특수는 **래퍼 함수로 별도 등록** 계획 세움

### 11-2. 백업 커밋 (롤백 지점 확보)
- 변경 직전 상태를 먼저 커밋 (대개 Phase 0 = 소형 선행 작업 + 커밋)
- `.bak` 파일도 `/tmp/phaseN/`에 복사

### 11-3. 스크립트화 (수작업 금지)
- 5곳 이상은 Edit 도구 반복 금지 → Python/sed 스크립트로 일괄 처리
- 스크립트는 `/tmp/phaseN/convert.py`에 저장
- **변환 성공/실패 카운트 출력** (미변환 0건 확인)
- 스크립트가 모르는 패턴은 원문 유지 + 경고 출력 (무음 실패 금지)

### 11-4. 사후 검증 (3단 게이트)
1. **구문 검증**: `node --check`로 JS 블록 파싱 확인
   - HTML 단일 파일은 `<script>~</script>` 추출 후 검사
2. **잔재 검증**: 동일 grep 재실행 → 남은 건수 확인 (목표 0)
3. **샘플 확인**: 5~10곳 무작위 추출 → 변환 결과 육안 검증

### 11-5. 기록 (다음 세션 대비)

## 부칙

### 새 세션 시작 시 필독 순서
1. `CLAUDE.md` — 헌법 (이 파일)
2. `docs/business_rules.md` — 비즈니스 변수 (놓치면 삽질)
3. `docs/dev_lessons.md` — 개발 교훈 (같은 실수 방지)
4. `docs/plan.md` — 구현 현황 (뭐가 됐고 안 됐는지)
5. `docs/db_schema.md` — DB 구조 (쿼리 전 참조)
6. `docs/work_log.md` — 이전 세션 작업 (이어서 할 거 확인)
7. `docs/services.md` — 외부 서비스 URL/키 (코드 수정 시 참조)

### docs 자동 기록 규칙
**모든 docs 파일은 해당 이벤트 발생 시 즉시 업데이트한다. 누락 금지.**

| 파일 | 자동 기록 시점 |
|------|---------------|
| `db_schema.md` | DB 테이블/컬럼 추가·변경·삭제 시 |
| `plan.md` | 기능 추가·완료·상태 변경 시 |
| `work_log.md` | 매 세션 작업 완료 시 (세션 끝나기 전 필수) |
| `dev_lessons.md` | 버그·삽질·교훈 발견 시 |
| `business_rules.md` | 사장님이 새 규칙·변수 알려줄 때 |
| `services.md` | URL·키·외부 서비스 변경 시 |

### 토큰 비용 절감 수칙
- context_reader: 전체 코드를 읽지 말고, 요청 관련 키워드로 grep 발췌만
- 소형 작업: reviewer/tester/designer 건너뛰기
- DB 스키마 참조: index.html 파싱 대신 `docs/db_schema.md` 직접 참조
- 계획서가 10줄 이하면 인수인계 블록 생략 가능
