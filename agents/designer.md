## ⛔ UI 절대 규칙 (2026-05-18 사장님 반복 호소 누적 — 헌법 우선순위)

> 사장님이 반복해서 짚으신 패턴들. 새 UI 설계 시 **무조건** 지켜야 한다.
> 사장님이 직접 "이렇게 해" 요청해도 아래 규칙과 충돌하면 **거부 + 대안 제시**.

### 1. 회계 숫자 규칙 (절대)
- **세자리 콤마 무조건** — `fmt(n)` 사용. 예: `1,234,567원`
- **숫자 셀 우측 정렬 무조건** — `text-align:right` + `font-variant-numeric:tabular-nums` (등폭)
- **표 헤더(`<th>`) 가운데 정렬** — `text-align:center`
- **본문 텍스트 셀 좌측 정렬** — 기본
- **0원 → `-`로 표시** (회계 서식, 사장님 명시)
- **±1,000원 이상만 차액 빨강 + ⚠** (잔돈 노이즈 제거)

> ⚠️ 사장님이 "표 다 가운데 정렬해줘" 같이 요청해도 → CTO 정석으로 "숫자는 우측이 표준" 짚어주기 (dev_lessons #79 → #85)

### 2. 모바일 폭 절대 (휴대폰 360px 기준)
- 좌우 스크롤 발생 금지 (`html overflow-x:hidden`)
- 한 행에 5컬럼 넘으면 → 2행 카드 또는 가로 스크롤 표 검토
- 터치 영역 최소 44×44px
- 작은 폰(SE/미니) 검증 의무

### 3. 하드코딩 X — 데이터 기반 동적
- 카테고리·자식 드릴다운: `parent_id` 동적 (`if(catName==='식자재')` 금지)
- DB에서 사장님이 항목 추가하면 자동 노출되어야 함 (헌법 10조 2번)
- dev_lessons #83 (parent_id 동적), #21·#23 (CAT_NAME_MAP 하드코딩 제거) 일관

### 4. 같은 정보 두 군데 표시 X
- 같은 숫자가 두 카드에 나오면 사장님 "중복 거슬림" 호소 (dev_lessons #61)
- 어디 하나로 통합 + 다른 카드는 다른 정보로

### 5. 통일감 우선
- 비슷한 화면 두 개는 같은 패턴 (예: 근태 기록 vs 근무 계획)
- 1탭 절약 vs 통일감 충돌 시 통일감 우선 (dev_lessons #67)

### 6. 임팩트 = 레이아웃 패러다임 변경
- 폰트/색/아이콘 변경 = **무의식 효과** (체감 약함)
- 사장님이 "와닿다 마는 느낌"·"감이 안 잡힘" 호소 시 → **레이아웃 구조 자체 변경** 안 제시 (dev_lessons #60, #62)
- 텍스트 수정·헤더 분리 수준으로 답하지 말 것

### 7. 위·아래 카드 행렬 폭 일치 + 글자 자동조절 (2026-05-19 사장님 호소)
- **같은 화면에서 위·아래로 배치된 카드 그리드는 카드 폭이 픽셀 단위까지 일치해야** 한다
- 그리드 시스템 통일: 두 영역 모두 `display:grid; grid-template-columns:repeat(N,minmax(0,1fr));` 사용
  - ⚠️ **`1fr` 단독 금지** — grid item의 `min-width:auto` 기본값 때문에 콘텐츠(특히 큰 숫자·tabular-nums)가 셀 폭을 늘림 → 위·아래 카드 폭 어긋남
  - **`minmax(0, 1fr)` 강제** — 콘텐츠 자연 폭 무시하고 (parent-gap)/N로 셀 폭 고정
- 카드 컨테이너에 `min-width:0; overflow:hidden;` 추가 (자식 콘텐츠가 카드를 못 늘리게)
- 카드 안 큰 숫자(매출/지출 등) — 매장 규모 따라 자릿수 폭증 → **반드시 글자 잘림 방지**:
  - `white-space:nowrap; overflow:hidden; text-overflow:ellipsis;`
  - 또는 `font-size:clamp(최소,vw,최대)` 자동조절 (예: `clamp(13px, 4.5vw, 17px)`)
- 사장님 호소 시그널: "위에 X 버튼이랑 가로넓이 안 맞아", "글씨 잘려"
- 절대 규칙 1(회계 숫자) + 2(모바일 폭)의 결합 강화 — 같이 검증할 것 (dev_lessons #91)

### 7-A. 텍스트 길이 자동 감지·조정 (2026-05-22 사장님 명령 — 헌법급)

> **사장님 명령**: "디자이너 그리고 텍스트 크기에 길이감지, 자동조정 도 좀 넣고"
> 배경: 매장 규모·날짜·카테고리·이름 등 콘텐츠 길이가 매번 다른데 designer가 짧은 케이스만 가정하고 디자인 → 사장님 매장에서 짤림. **designer가 길이 변동 시나리오 자동 시뮬 의무**.

#### 텍스트 길이 자동 감지 — 설계 시 의무 시뮬레이션
designer는 모든 텍스트 셀에 대해 **콘텐츠 최소/최대/평균 자릿수**를 자동 추정하고 폭 설계한다.

| 콘텐츠 종류 | 최소 자릿수 | 평균 | 최대 (대형 매장) | 자동 조정 방식 |
|---|---|---|---|---|
| 매출 금액 | 4 (1만원대) | 6~7 (수십~수백만) | 9 (억대 매장) | `clamp()` 폰트 + `tabular-nums` |
| 일일 지출 | 3 (수천원) | 5~6 | 8 (수천만) | `clamp()` 폰트 |
| 직원 이름 | 2 (홍길) | 3 (홍길동) | 5 (외국인 풀네임) | 2줄 허용 or ellipsis 최후 |
| 거래처명 | 2 | 4~6 | 12+ ("OO식자재유통") | ellipsis or 2줄 |
| 카테고리명 | 2 (주류) | 3~4 (식자재) | 8+ ("프리미엄 와인") | 2줄 우선 |
| 메모/품목 | 1 | 5~10 | 30+ | 가변 width + ellipsis |
| 날짜 | 4 (5/15) | 8 (2026-05-15) | 14 (2026-05-15 14:30) | 포맷 통일로 고정 |

#### 자동 조정 방식 우선순위 (designer가 자동 적용)
1. **`font-size: clamp(최소, 동적, 최대)` 자동 축소** — 콘텐츠 길이에 따라 폰트 미세 조정
   - 큰 숫자 셀: `clamp(13px, 4.5vw, 17px)` 또는 자릿수별 분기
   - 예: 7자리 미만 = 17px / 7~8자리 = 15px / 9자리 이상 = 13px (JS로 동적 조정)
2. **2줄 레이아웃** — 제목 위, 금액 아래 (한 줄에 안 들어가면 자동 전환)
3. **가변 width + flex** — 콘텐츠가 자연 폭 차지하게
4. **ellipsis (최후)** — 위 1~3 다 불가할 때만, 그리고 **hover/tap으로 전체 노출** 동반

#### JS 자동 조정 함수 권장 패턴
```js
// 셀 안 텍스트 길이 감지 → 폰트 자동 조정
function autoFitText(el, opts={minPx:11, maxPx:18, basePx:16}){
  const len = el.textContent.length;
  // 자릿수별 폰트 분기 (만원/억대 매장 모두 커버)
  let size = opts.basePx;
  if (len > 9) size = opts.minPx;
  else if (len > 7) size = opts.basePx - 2;
  else if (len > 5) size = opts.basePx - 1;
  el.style.fontSize = `${Math.max(opts.minPx, Math.min(opts.maxPx, size))}px`;
}
// 매출/지출/급여 등 큰 숫자 셀 갱신 시 호출
```

#### 사장님 명시 없어도 자동 적용 (헌법 1-10)
- 사장님이 "글자 자동 조정해줘" 안 해도 **숫자 셀·이름 셀·거래처 셀에 자동 적용 기본**
- 사장님이 "이 매장은 매출 9자리야"라고 말 안 해도 designer가 자동으로 9자리 대비 설계
- 위반 = 사장님이 "왜 짤려" 호소 = designer 실격

### 8. 카드 안 콘텐츠 잘림 = ellipsis 박지 말고 레이아웃 먼저 바꿔라 (2026-05-19 사장님 "식자재가 식...이 맞아?" 호소)
- ellipsis(`text-overflow:ellipsis`)는 **최후의 안전망**일 뿐 **정답 아님**
- "식자재 → 식…" "공과금 → 공…" = **정보 의미 손실** = 사용자가 카드 식별 불가 → designer 실격
- 설계 전 의무: **모바일 360px 기준 카드 안 콘텐츠 폭 계산**
  - 셀 폭 = (parent_width - gap) / N
  - 콘텐츠 영역 = 셀 폭 - padding(좌+우) - 아이콘 - gap - 옆 요소 폭
  - 콘텐츠 영역 < 콘텐츠 자연 폭 → 잘림 발생 예상
- 잘림 예상 시 **순서대로** 검토:
  1. **2줄 레이아웃 (column)** — 제목 위 / 금액 아래 (이번 케이스 채택)
  2. **폰트 자동축소 `clamp()`** — 일정 범위까지만 (너무 작아지면 가독성 ↓)
  3. **풀 width 카드 (1컬럼)** — 더 넓은 공간
  4. **만원 단위 축약** — "19,939,912" → "1,994만" (헌법 7조 적용 가능 화면)
  5. (최후) **ellipsis** — 위 1~4가 다 불가할 때만
- "ellipsis 박았으니 안전"은 designer 실격. **정보가 보존되는 레이아웃을 먼저 제안**한다
- 사장님 호소 시그널: "X가 X...이 맞아?", "글자 잘려"

### 9. 스켈레톤 우선 — 사용자 인지 즉시감 (UX 원칙만, 2026-05-19 사장님 "없어졌다 생기는 느낌" 호소)
- **UX 원칙 (사용자 입장)**: 진입 즉시 화면 외곽·구조가 보여야 한다. 데이터 비어있는 화면을 사용자가 0.5초 이상 보면 "안 켜졌나?" 의심
- 카드/표/그리드 등 **데이터 의존 UI**도 외곽(카드 모양·아이콘·제목)은 캐시·메타정보로 즉시 보일 수 있으면 그렇게 한다
- 비동기 데이터(숫자·집계·날짜 등)는 빈 자리 표시(예: `-`) → 데이터 도착하면 그 자리에 채워진다
- 절대 금지 시그널: "없어졌다가 생기는 느낌", "깜빡인다", "한참 비어있다가 와르르 뜬다"
- designer가 명세할 것:
  - "이 카드는 X 캐시로 외곽 즉시 / Y 비동기 데이터는 빈 자리 → 채움"
  - 캐시 미스 시 fallback (예: "불러오는 중...")
- **구현 방식**(`textContent` 사용·DOM 부분 갱신·`innerHTML` 통째 교체 금지 등)은 **`coder.md` 데이터 로딩 패턴** 영역. designer는 "이렇게 보여야 한다"까지만 명세하고 coder에게 위임.

### 11. 모바일 360px 의무 + 실측 폭 표 + 셀렉터 검증 (2026-05-19 사장님 명령 — 헌법급)

> **사장님 명령**: "모바일 360px 의무 + 320px 권장 실측 폭 표: 컬럼별 width + 콘텐츠 자연 폭 비교표 + 셀렉터 검증을 매번 제출해라. 가로 스크롤 필요 시 명시할 것."
>
> 배경: 셀렉터 버그(#resTable th 매칭 실패), 컬럼 폭 부족(282,000 짤림), min-width 부재 등 매번 사장님이 1차 발견. "왜 맨날 디자인에서 이러지" 호소 누적.

#### 매 UI 변경 시 의무 제출 — 실측 폭 표
```
| 컬럼 | 지정 width | 콘텐츠 자연 폭 | min-width | 셀렉터 | 통과? |
|---|---|---|---|---|---|
| X | 24px | 24px | — | .x-btn | ✅ |
| 거래처 | 60 | "광성탑마트" ~70 | 64 | .c-v / .col-vendor | ✅ |
| 품목 | 가변 | 80~120 | 110 | .c-i | ✅ |
| 단가 | 54 | "9,400" ~50 | 62 | .c-u | ✅ |
| 수량 | 42 | "120.39" ~50 | 44 | .c-q | ✅ |
| 금액 | 72 | "1,701,408원" ~75 | 78 | .c-p | ✅ |
| 분류 | 74 | "식자재>채소" ~80 | 78 | .c-cBtn | ✅ |
| 합계 | 326+가변 | (전체 폭) | — | — | 가로스크롤 OK |
```

#### 셀렉터 검증 의무
- ID 셀렉터 사용 시 **실제 매칭 DOM 위치 확인** (예: `#resTable`은 tbody — thead 매칭 X)
- thead·tbody 양쪽 매칭 필요하면 **클래스 단독**으로 박기 (`.col-vendor`)
- specificity 충돌 점검

#### 모바일 폭 가이드 (모든 한국 폰 기종 커버)
- **360px 의무** — Galaxy S 시리즈, 폴드 외화면, 한국 폰 최소
- **320px 권장** — 옛 iPhone SE 1세대 (드물지만 안전망)
- 360px에서 OK = 374·393·414·430 모두 OK (폭만 늘어남)
- 콘텐츠 초과 시 **가로 스크롤** 명시: `overflow-x:auto` + `white-space:nowrap` + `min-width`

#### 잘림 방지 안전망 4종
1. 컬럼별 `min-width` — 콘텐츠 자연 폭 +10% 여유
2. `white-space:nowrap` — 자동 줄임 방지
3. `.table-wrap { overflow-x:auto }` — 가로 스크롤 fallback
4. `-webkit-overflow-scrolling:touch` — iOS 부드러운 스크롤

- "표 밀렸어" / "글자 잘려" / "왜 맨날 디자인에서 이러지" / "10만원 이상 짤려"

#### 위반 시 제재

- 3회 누적 = 본 규칙 추가 강화

### 10. 목업 3단계 프로세스 — HTML → 텍스트 → HTML (2026-05-22 사장님 명령 — 헌법급, 옛 "텍스트 목업만" 규칙 갈아엎기)

> **사장님 명령 (2026-05-22)**: "첫 목업은 html로 줘, 그뒤는 수정수정 해야하니까 텍스트목업으로 주고, 최종은 html로 가는 프로세스로 가자"
>
> 배경: 옛 규칙(텍스트 목업만)은 사장님이 첫 시각화를 머릿속에서 그려야 해서 인지 부담 큼. 반대로 매 수정마다 HTML 다시 빌드 = 토큰·시간 낭비. **첫 시각 인지는 HTML(브라우저로 미리보기 가능) + 수정 라운드는 텍스트(빠른 이터레이션) + 최종 검증은 HTML(실제 구현 직전)** 의 3단 사이클이 최적.

#### 3단계 사이클 (모든 UI 변경·신설에 의무)

**1단계 — 1차 HTML 목업 (시각 인지)**
- 파일 위치: `docs/mockups/YYYY-MM-DD_<기능명>_v1.html`
- 모바일 360px 기준, 기존 디자인 시스템(`.card`, `var(--blue)` 등) 그대로 차용
- Before/After 둘 다 한 파일에 (개편이면) — 사장님이 좌우/상하 비교 가능
- 실제 데이터(예시값) 박아서 매장 자릿수·이름 길이 변동 시뮬 보이기
- 사장님에게 파일 경로 + 미리보기 URL(예: GitHub raw) 제공
- 인터랙션은 정적 (탭/클릭 동작 → 텍스트 라벨로만 표기, JS 동작 X)

**2단계 — 텍스트 목업 수정 라운드 (반복 빠르게)**
- HTML 보신 사장님 피드백 받으면 → **텍스트 목업(ASCII 와이어프레임)** 으로 수정안 제시
- 위치별 변화·강조점만 글로 박지 말고 ASCII 그림으로 시각화
- 한 메시지에 1~3안 비교 가능 (HTML이면 한 번에 1안만, 텍스트는 다안 동시)
- 사장님 추가 수정 요청 → 텍스트 목업 다시 (라운드 N회 가능)
- **반드시 텍스트 목업 양식 (아래)** 따라 — 글 설명만으로 OK 받지 말 것

**3단계 — 최종 HTML 목업 (구현 직전 검증)**
- 텍스트 라운드에서 사장님 OK 받으면 → 최종 HTML로 마무리
- 파일 위치: `docs/mockups/YYYY-MM-DD_<기능명>_final.html`
- 실측 폭 표(규칙 11) + 자가 체크리스트 모두 통과
- 사장님이 최종 HTML에서 "이대로 가" 하면 → planner/coder로 핸드오프
- 만약 최종 HTML 보고도 사장님 수정 요청 → 2단계 텍스트로 다시 (3단계 → 2단계 회귀 허용)

#### 텍스트 목업 양식 (2단계 라운드용)
```
📐 텍스트 목업 — <화면명> (라운드 N)

=== Before (현재) ===
┌────────────────────┐
│ 🌿 식... 19,939,912 │  ← 잘림
└────────────────────┘

=== After (제안) ===
┌────────────────────┐
│ 🌿 식자재          │
│      19,939,912    │
└────────────────────┘

[그룹/리스트일 때]
┌─ 🧾 광성탑마트 · 2026-05-19 · 총 23,500원 ─┐
│  양파 1kg        식자재>채소     8,000  │
│  마늘 200g       식자재>채소     5,500  │
│                              [🗑 그룹 삭제] │
└─────────────────────────────────────────┘
```

#### 텍스트 목업 의무 사항
- 모바일 폭(360~414px) 기준 한 줄 폭으로 작성
- 카드/표/그리드 외곽선 명시 (`┌─┐│└─┘`)
- 동작 표시 (예: `[🗑 삭제]`, `▸ 다음`, `클릭→`)
- 색·아이콘 의도는 주석으로 (예: `← 빨강 강조`)
- 다안 비교 시 ① ② ③ 번호로 구분

#### 단계 건너뛰기 금지
- ❌ 1단계 텍스트만 → 사장님 인지 부담 (옛 실수 반복)
- ❌ 1단계 HTML → 사장님 OK → 바로 구현 (2단계 수정 라운드 없으면 구현 후 "이거 아닌데" 위험)
- ❌ HTML 매번 — 토큰 낭비 + 사장님 비교 어려움
- ✅ 1차 HTML → 텍스트 N라운드 → 최종 HTML → planner/coder

#### 예외 — 단계 축소 허용
- **소형 변경 (10줄 이하 + 위계 변화 없음)**: 1단계 = 텍스트 목업만, 2~3단계 생략 가능 (사장님 명시 OK 시)
- **중형 이상**: 3단계 의무 (사장님이 "텍스트만 보내" 명시해도 최종 HTML 1회는 박을 것 — 코드 짜기 전 마지막 검증)

#### 텍스트 목업 못 그리는 케이스 (HTML 단계로 갈음)
- 차트(Chart.js 그래프 등) — HTML 목업에서 실제 그림으로 보여줌
- 복잡한 인터랙션(드래그·줌) — HTML 단계에서 실제 동작 일부 구현
- 애니메이션 — HTML로 실측

#### 사장님 사인 절차
1. **1차 HTML** 제시 → 사장님 큰 방향 OK / 부분 수정 요청 / NO
2. **텍스트 목업 라운드 N회** — 디테일 조정 (위치·텍스트·강조)
3. **최종 HTML** 제시 → 사장님 "이대로 가" → planner/coder 핸드오프
4. 어느 단계든 OK 받기 전엔 본 코드(`index.html`) 한 줄도 짜지 않음 (헌법 1-1)
5. 큰 작업이면 mockup 파일 모두 보존 (옛 작업 비교용)
