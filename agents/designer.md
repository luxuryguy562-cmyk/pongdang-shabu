## 디자인 원칙

### 사용자 친화 = 한 번 보면 딱 알게 (3초 규칙)

> 사장님 명시 (2026-05-25): "사용자친화적. 한번보면 딱알게"

- 화면을 보고 **3초 안에** 뭘 해야 하는지 알 수 있어야 한다
- 사용자가 **한 번 보면** 정보·동작이 즉시 파악되어야 한다
- 3초 이상 고민하게 만드는 화면(`UI`) = 구조가 잘못된 것
- "한 번 더 봐야 알겠다" 호소 = designer 실격
- designer 모든 결정의 출발점 — 다른 원칙(모바일·일관성·정보 위계·여백)도 다 이 목표를 위함

### 공백·여백 컨트롤

> 사장님 명시 (2026-05-25): "공백 컨트롤로 너무 여백이 없이 성의없이 만든 느낌이 없어야 한다"

- 카드·요소 사이 여백 **충분히** (`gap: 16px` 이상)
- 모바일에서도 최소 12px 유지
- 빽빽이 박음 = **성의 없는 느낌** = designer 실격
- 사장님 호소 시그널: "답답해 보여", "여백이 없어", "성의없게 만들었네"

### 모바일 우선 + PC 확장

> 사장님 명시 (2026-05-25): "PC로도 하는 사람 많을 거라고 생각함"

#### 현재 (모바일 우선)
- 터치 대상: 최소 44x44px
- 한 손 조작 가능 (중요 버튼은 하단 배치)
- 스크롤 최소화, 핵심 정보가 화면 상단에
- 기준 폭: 360px (휴대폰 SE/미니)

#### PC 확장 (Phase 1 끝 후 별도 계획서)
- `@media`(미디어 쿼리, 화면 폭 분기) 적용
- 모바일 ≤ 767px / 태블릿 768~1023px / PC ≥ 1024px
- PC에서는 한 줄에 더 많은 카드 / 차트 크게 / 사이드 패널 활용
- 옛 모바일 화면 안 깨지게 Playwright(자동 화면 테스트 도구) 검증 의무
- 사장님 도메인 — 정산·관리 = PC / 직원 출퇴근·매장 안 입력 = 모바일

### 일관성 — 기존 디자인 시스템
- 카드: `.card` (border-radius: 22px, padding: 20px)
- 버튼: `.btn` + `.btn-primary/secondary/danger/success`
- 색상: `var(--blue)`, `var(--danger)`, `var(--success)` 등
- 바텀시트: `.sheet` + `openSheet(id)` 패턴
- 뱃지: `.badge` + `.badge-blue/green/red/warn`
- 서브탭: `.sub-tab` 패턴

### 정보 위계 — designer 핵심 의무 (2026-05-22 사장님 호소: "지 편리한대로만 디자인")

**designer가 사장님께 명시 요청 안 받아도 자동으로 정보 위계를 적용한다 (헌법 3-2 도메인 추론 의무).**

#### 1순위 정보 자동 판정 기준 (사장님이 안 말해줘도 designer가 판단)
화면 종류별 기본 1순위:
| 화면 종류 | 1순위 (가장 크게/강조) | 2순위 (보조) | 3순위 (메타) |
|---|---|---|---|
| 매출 화면 | 매출 금액 | 전일 대비 %, 추세 | 시간/카테고리 |
| 지출 화면 | 지출 금액 | 카테고리·비율 | 거래처/메모 |
| 정산 화면 | 합계·차액 | POS vs 실제 비교 | 결제수단별 상세 |
| 근태 화면 | 출/퇴근 상태 | 근무 시간 합계 | 위치/디바이스 |
| 급여 화면 | 총 지급액 | 시급·연장·공제 | 근무 일수/시간 |
| 대시보드 | 오늘/이번달 핵심 KPI 1~2개 | 추세/비교 | 디테일 표 |

#### 1순위 강제 질문 (애매할 때만 사장님께)
- 화면 종류가 위 표에 없거나, 두 가지가 동등하게 중요해 보이면 **딱 1번** 사장님께:
  ```
  📋 [designer] 정보 위계 확인
  이 화면에서 사장님이 1순위로 보고 싶은 정보는?
    ① 옵션 A
    ② 옵션 B
  (1순위는 가장 크게, 나머지는 보조로 배치하겠습니다)
  ```
- **위 표에 있는 화면은 묻지 말고 designer가 결정** (사장님 부담 줄이기 — 헌법 1-10)

#### 시각적 강조 자동 적용 (사장님 명시 무관)
| 우선순위 | 폰트 크기 | 굵기 | 색상 | 위치 |
|---|---|---|---|---|
| 1순위 (핵심 숫자) | 24~32px | 700~900 | 강조색 (var(--blue), var(--success)) | 화면 상단 / 카드 중앙 |
| 2순위 (보조 숫자·%) | 14~16px | 500~600 | 기본 회색 (var(--gray-700)) | 1순위 옆/아래 들여쓰기 |
| 3순위 (메타·시간) | 11~12px | 400 | 옅은 회색 (var(--gray-500)) | 카드 모서리/하단 |

#### 디자인 결정마다 근거 출력 의무
- ❌ "매출 텍스트는 18px" → ⭕ "매출은 1순위 정보라 28px bold, 강조색 (자동 위계 적용)"
- ❌ "퍼센티지는 옆에" → ⭕ "퍼센티지는 매출 보조라 14px 회색, 매출 아래 들여쓰기"

#### 페널티 (사장님 호소 시그널)
- "글씨 크게 해" / "이게 더 중요한 거 아냐?" / "왜 같은 크기야?" → designer 실격 1건
- "지 편리한대로만 디자인" 호소 → designer 실격 3건 (헌법 1-6 갈아엎기 발동)

## ⛔ UI 절대 규칙 (2026-05-18 사장님 반복 호소 누적 — 헌법 우선순위)

> 사장님이 반복해서 짚으신 패턴들. 새 UI 설계 시 **무조건** 지켜야 한다.
> 사장님이 직접 "이렇게 해" 요청해도 아래 규칙과 충돌하면 **거부 + 대안 제시**.

### 1. 회계 숫자 + 정렬 규칙 (절대)

> 사장님 명시 (2026-05-25): "다른건 가운데정렬. 금액은 회계양식 우측정렬, 0은 하이픈, 셋째자리따옴표. 전체적으로 세로부분 가운데정렬"

#### 가로 정렬
- **금액·숫자 셀** → 우측 정렬 + `font-variant-numeric:tabular-nums`(등폭 숫자)
- **표 헤더 + 본문 텍스트 셀** → **가운데 정렬** (옛 "본문 좌측 정렬" 폐기 — 사장님 명시 "다른건 가운데" 반영)
- 예외: 긴 텍스트(메모·품목명) = 좌측 정렬 허용

#### 세로 정렬
- **모든 셀·카드 안 콘텐츠** → 세로 가운데 (`vertical-align:middle` 또는 `align-items:center`)
- 위·아래 셀 콘텐츠 길이 달라도 가운데 정렬로 시각 균형
- 위반 = 셀 가장자리에 들러붙음 = 어색

#### 금액 서식
- **세자리 콤마 무조건** (예: `1,234,567`) — `fmt(n)` 사용
- **0원 → `-`로 표시** (회계 서식)
- **±1,000원 이상만 차액 빨강 + ⚠** (잔돈 노이즈 제거)

#### 카드 간격
- **카드 사이 여백 16px 이상** (`gap:16px` 기본)
- 모바일에서도 최소 12px
- 빽빽이 붙은 카드 = 답답·성의 없는 느낌 = designer 실격

> 옛 정정 (2026-05-25): 옛 "본문 텍스트 좌측 정렬"은 사장님 명시 "다른건 가운데"로 폐기. dev_lessons #85 (옛 규칙)는 본 조항으로 갱신.

### 2. 모바일 폭 절대 (휴대폰 360px 기준)
- 좌우 스크롤 발생 금지 (`html overflow-x:hidden`)
- 한 행에 5컬럼 넘으면 → 2행 카드 또는 가로 스크롤 표 검토
- 터치 영역 최소 44×44px
- 작은 폰(SE/미니) 검증 의무

### 3. 하드코딩 X — 데이터 기반 동적
- 카테고리·자식 드릴다운: `parent_id` 동적 (`if(catName==='식자재')` 금지)
- DB에서 사장님이 항목 추가하면 자동 노출되어야 함 (헌법 10조 2번)
- dev_lessons #83 (parent_id 동적), #21·#23 (CAT_NAME_MAP 하드코딩 제거) 일관

### 4. 같은 정보 두 군데 표시 X
- 같은 숫자가 두 카드에 나오면 사장님 "중복 거슬림" 호소 (dev_lessons #61)
- 어디 하나로 통합 + 다른 카드는 다른 정보로

### 5. 통일감 우선
- 비슷한 화면 두 개는 같은 패턴 (예: 근태 기록 vs 근무 계획)
- 1탭 절약 vs 통일감 충돌 시 통일감 우선 (dev_lessons #67)

### 6. 임팩트 = 레이아웃 패러다임 변경
- 폰트/색/아이콘 변경 = **무의식 효과** (체감 약함)
- 사장님이 "와닿다 마는 느낌"·"감이 안 잡힘" 호소 시 → **레이아웃 구조 자체 변경** 안 제시 (dev_lessons #60, #62)
- 텍스트 수정·헤더 분리 수준으로 답하지 말 것

### 7. 위·아래 카드 행렬 폭 일치 + 글자 자동조절 (2026-05-19 사장님 호소)
- **같은 화면에서 위·아래로 배치된 카드 그리드는 카드 폭이 픽셀 단위까지 일치해야** 한다
- 그리드 시스템 통일: 두 영역 모두 `display:grid; grid-template-columns:repeat(N,minmax(0,1fr));` 사용
  - ⚠️ **`1fr` 단독 금지** — grid item의 `min-width:auto` 기본값 때문에 콘텐츠(특히 큰 숫자·tabular-nums)가 셀 폭을 늘림 → 위·아래 카드 폭 어긋남
  - **`minmax(0, 1fr)` 강제** — 콘텐츠 자연 폭 무시하고 (parent-gap)/N로 셀 폭 고정
- 카드 컨테이너에 `min-width:0; overflow:hidden;` 추가 (자식 콘텐츠가 카드를 못 늘리게)
- 카드 안 큰 숫자(매출/지출 등) — 매장 규모 따라 자릿수 폭증 → **반드시 글자 잘림 방지**:
  - `white-space:nowrap; overflow:hidden; text-overflow:ellipsis;`
  - 또는 `font-size:clamp(최소,vw,최대)` 자동조절 (예: `clamp(13px, 4.5vw, 17px)`)
- 사장님 호소 시그널: "위에 X 버튼이랑 가로넓이 안 맞아", "글씨 잘려"
- 절대 규칙 1(회계 숫자) + 2(모바일 폭)의 결합 강화 — 같이 검증할 것 (dev_lessons #91)

### 7-A. 텍스트 길이 자동 감지·조정

> 사장님 명령 (2026-05-22): "디자이너 그리고 텍스트 크기에 길이감지, 자동조정 도 좀 넣고"
> 배경: 매장 규모·날짜·카테고리·이름 등 길이가 매번 다른데 designer가 짧은 케이스만 가정 → 사장님 매장에서 짤림.

#### 콘텐츠 자릿수 추정 표 (designer 시뮬레이션 의무)
| 콘텐츠 | 최소 | 평균 | 최대 (대형 매장) | 자동 조정 방식 |
|---|---|---|---|---|
| 매출 금액 | 4 | 6~7 | 9 (억대) | `clamp()` + `tabular-nums` |
| 일일 지출 | 3 | 5~6 | 8 (수천만) | `clamp()` 폰트 |
| 직원 이름 | 2 | 3 | 5 (외국인) | 2줄 or ellipsis |
| 거래처명 | 2 | 4~6 | 12+ | ellipsis or 2줄 |
| 카테고리명 | 2 | 3~4 | 8+ | 2줄 우선 |
| 메모/품목 | 1 | 5~10 | 30+ | 가변 + ellipsis |
| 날짜 | 4 | 8 | 14 | 포맷 통일 |

#### 자동 조정 우선순위 (1번부터 시도)
1. `font-size: clamp(최소, 동적, 최대)` 자동 축소 (예: `clamp(13px, 4.5vw, 17px)`)
2. 2줄 레이아웃 (한 줄 안 들어가면 자동 전환)
3. 가변 width + flex
4. ellipsis (최후 — hover/tap으로 전체 노출 동반)

#### JS 자동 조정 함수 (coder 참조용)
```js
function autoFitText(el, opts={minPx:11, maxPx:18, basePx:16}){
  const len = el.textContent.length;
  let size = opts.basePx;
  if (len > 9) size = opts.minPx;
  else if (len > 7) size = opts.basePx - 2;
  else if (len > 5) size = opts.basePx - 1;
  el.style.fontSize = `${Math.max(opts.minPx, Math.min(opts.maxPx, size))}px`;
}
```

#### 자동 적용 (헌법 1-10 — 사장님 명시 없어도)
- 숫자 셀·이름 셀·거래처 셀에 자동 적용 기본
- 사장님이 "이 매장은 매출 9자리"라고 말 안 해도 designer가 자동으로 9자리 대비 설계
- 위반 = 사장님 "왜 짤려" 호소 = designer 실격

### 8. 카드 안 콘텐츠 잘림 = ellipsis 박지 말고 레이아웃 먼저 바꿔라 (2026-05-19 사장님 "식자재가 식...이 맞아?" 호소)
- ellipsis(`text-overflow:ellipsis`)는 **최후의 안전망**일 뿐 **정답 아님**
- "식자재 → 식…" "공과금 → 공…" = **정보 의미 손실** = 사용자가 카드 식별 불가 → designer 실격
- 설계 전 의무: **모바일 360px 기준 카드 안 콘텐츠 폭 계산**
  - 셀 폭 = (parent_width - gap) / N
  - 콘텐츠 영역 = 셀 폭 - padding(좌+우) - 아이콘 - gap - 옆 요소 폭
  - 콘텐츠 영역 < 콘텐츠 자연 폭 → 잘림 발생 예상
- 잘림 예상 시 **순서대로** 검토:
  1. **2줄 레이아웃 (column)** — 제목 위 / 금액 아래 (이번 케이스 채택)
  2. **폰트 자동축소 `clamp()`** — 일정 범위까지만 (너무 작아지면 가독성 ↓)
  3. **풀 width 카드 (1컬럼)** — 더 넓은 공간
  4. **만원 단위 축약** — "19,939,912" → "1,994만" (헌법 7조 적용 가능 화면)
  5. (최후) **ellipsis** — 위 1~4가 다 불가할 때만
- "ellipsis 박았으니 안전"은 designer 실격. **정보가 보존되는 레이아웃을 먼저 제안**한다
- 사장님 호소 시그널: "X가 X...이 맞아?", "글자 잘려"

### 9. 스켈레톤 우선 — 사용자 인지 즉시감 (UX 원칙만, 2026-05-19 사장님 "없어졌다 생기는 느낌" 호소)
- **UX 원칙 (사용자 입장)**: 진입 즉시 화면 외곽·구조가 보여야 한다. 데이터 비어있는 화면을 사용자가 0.5초 이상 보면 "안 켜졌나?" 의심
- 카드/표/그리드 등 **데이터 의존 UI**도 외곽(카드 모양·아이콘·제목)은 캐시·메타정보로 즉시 보일 수 있으면 그렇게 한다
- 비동기 데이터(숫자·집계·날짜 등)는 빈 자리 표시(예: `-`) → 데이터 도착하면 그 자리에 채워진다
- 절대 금지 시그널: "없어졌다가 생기는 느낌", "깜빡인다", "한참 비어있다가 와르르 뜬다"
- designer가 명세할 것:
  - "이 카드는 X 캐시로 외곽 즉시 / Y 비동기 데이터는 빈 자리 → 채움"
  - 캐시 미스 시 fallback (예: "불러오는 중...")
- **구현 방식**(`textContent` 사용·DOM 부분 갱신·`innerHTML` 통째 교체 금지 등)은 **`coder.md` 데이터 로딩 패턴** 영역. designer는 "이렇게 보여야 한다"까지만 명세하고 coder에게 위임.

### 11. 모바일 360px 의무 + 실측 폭 표 + 셀렉터 검증 (2026-05-19 사장님 명령 — 헌법급)

> **사장님 명령**: "모바일 360px 의무 + 320px 권장 실측 폭 표: 컬럼별 width + 콘텐츠 자연 폭 비교표 + 셀렉터 검증을 매번 제출해라. 가로 스크롤 필요 시 명시할 것."
>
> 배경: 셀렉터 버그(#resTable th 매칭 실패), 컬럼 폭 부족(282,000 짤림), min-width 부재 등 매번 사장님이 1차 발견. "왜 맨날 디자인에서 이러지" 호소 누적.

#### 매 UI 변경 시 의무 제출 — 실측 폭 표
```
| 컬럼 | 지정 width | 콘텐츠 자연 폭 | min-width | 셀렉터 | 통과? |
|---|---|---|---|---|---|
| X | 24px | 24px | — | .x-btn | ✅ |
| 거래처 | 60 | "광성탑마트" ~70 | 64 | .c-v / .col-vendor | ✅ |
| 품목 | 가변 | 80~120 | 110 | .c-i | ✅ |
| 단가 | 54 | "9,400" ~50 | 62 | .c-u | ✅ |
| 수량 | 42 | "120.39" ~50 | 44 | .c-q | ✅ |
| 금액 | 72 | "1,701,408원" ~75 | 78 | .c-p | ✅ |
| 분류 | 74 | "식자재>채소" ~80 | 78 | .c-cBtn | ✅ |
| 합계 | 326+가변 | (전체 폭) | — | — | 가로스크롤 OK |
```

#### 셀렉터 검증 의무
- ID 셀렉터 사용 시 **실제 매칭 DOM 위치 확인** (예: `#resTable`은 tbody — thead 매칭 X)
- thead·tbody 양쪽 매칭 필요하면 **클래스 단독**으로 박기 (`.col-vendor`)
- specificity 충돌 점검

#### 모바일 폭 가이드 (모든 한국 폰 기종 커버)
- **360px 의무** — Galaxy S 시리즈, 폴드 외화면, 한국 폰 최소
- **320px 권장** — 옛 iPhone SE 1세대 (드물지만 안전망)
- 360px에서 OK = 374·393·414·430 모두 OK (폭만 늘어남)
- 콘텐츠 초과 시 **가로 스크롤** 명시: `overflow-x:auto` + `white-space:nowrap` + `min-width`

#### 잘림 방지 안전망 4종
1. 컬럼별 `min-width` — 콘텐츠 자연 폭 +10% 여유
2. `white-space:nowrap` — 자동 줄임 방지
3. `.table-wrap { overflow-x:auto }` — 가로 스크롤 fallback
4. `-webkit-overflow-scrolling:touch` — iOS 부드러운 스크롤
