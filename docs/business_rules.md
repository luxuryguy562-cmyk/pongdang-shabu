### 0-1. 대분류

#### 매출
- 마감 매출 = **현금 + 현금영수증 + 신용카드 + 기타결제** (4칸 합 — 9번 박힘)

- **가마감** = POS(가게 매출 통로) 입력 = 사장님 직접 박은 영업 매출
- **진마감** = 통장 입금 실제 금액 (mydata 통장 분류 — dev_lessons #99 그대로 유지)

#### 지출

#### 절대 금지 (Claude 헌법 1-7 추측 금지 일관)
- ❌ "사장님 = owner = 이송은" 한 덩어리로 묶기 — **김은성과 이송은은 다른 사람**
- ❌ owner 권한 = 어플 주인 권한으로 추측 — **owner는 매장 단위 권한이지 시스템 권한 아님**
- ❌ "앱 사용자 = 김은성"으로 옛날 표현 그대로 쓰기 — **현재 김은성은 사용자가 아니라 주인**
- ❌ admin 모드/시스템 권한을 `auth_level='owner'`로 게이트하기 — **김은성은 employees에 없어서 auth_level 자체가 없음**

#### Claude가 새 기능 만들 때 점검
- "사장님만 볼 수 있게" → **김은성만 (어플 주인)**. owner 권한이 아님. 별도 식별자 필요 (auth.users.email 또는 hardcoded secret)
- "owner만" → **각 매장의 매장 사장** (이송은처럼). 매장 단위 게이트. 다른 매장 데이터 X
- "관리자 대시보드" → **김은성 전용**. owner 게이트로 만들면 외부 매장 사장도 진입 가능 — 잘못된 설계

- 가입 화면 owner 입력 라벨: **"사장 이름"**. ("사장님" 존칭 금지, "총관리자" 표기 금지)

## 8. 영업개시 보고 + 차감 추적 (2026-05-12)

**배경**: 직원이 마감보고 시 매출/금고를 적게 적어 차액 0으로 맞춰 돈을 빼돌릴 위험. 또는 마감 후 누군가 금고에서 가져갈 위험.

### B안: 익일 영업개시로 검증
- 아침 출근 시 금고 다시 세서 시스템 입력 → 어제 마감과 비교
- 마감 차액(영업 책임) + 영업개시 차액(키 소지자 책임) 분리 추적

### 마감정산/영업개시 차감 구조 통일
- 두 화면 모두 동적 행 (type 셀렉트 + 금액 + 메모 + 행 추가/삭제)
- 기본 2행: 통장입금 / 기타사용 자동 표시
- 메모 = 사유 기록 (예: "사장님 출금", "직원 식대")

### 차액 의미
| 차액 | 정상 (0) | 불일치 | 의미 |
|---|---|---|---|
| **마감 차액** | 매출 = 금고 | 영업 중 어긋남 | 영업 책임 (직원/매장) |
| **영업개시 차액** | 어제 마감 − 차감 = 오늘 실제 | 차감 처리 후에도 어긋남 | **진짜 사라진 돈** (도난·실수) |

### 차감 처리 정책
- 마감 후 사장님 출금 → 직원이 다음 영업개시에서 차감으로 보고 + 메모
- 정상 운영이면 영업개시 차액 항상 0
- 차액 발생 시 = 사장님 조사 필요

### 차액 통합 추적
- 마감정산 → 기록 조회: 일별 표 (영업개시 / 마감 / 이상 발생 합)
- 이상 발생 합 = `|영업개시 차액| + |마감 차액|` (절댓값 합)
- 이번달 누적 → "이번달 X원 어디선가 새고 있다" 한눈에

## 9. 매출 = 마감 4칸 (2026-05-12 확립)

**규칙**: 마감 매출은 `현금 + 현금영수증 + 신용카드 + 기타결제` 4칸 합만으로 계산.
- 현금 상세 3칸 (현금/QR/이체) = `pos_cash` 분해 정보. 매출 합산에 **포함 X**.
- `sales_daily` 동기화: `cash=pos_cash`, `qr=0`, `etc=pos_etc` (현금상세 빼고).

**과거 문제**:
- 옛 매핑은 `cash=cash_detail_cash, qr=cash_detail_qr, etc=pos_etc+cash_detail_transfer` → 직원이 매출 4칸 안 채우고 현금상세만 채우면 가짜 매출 잡힘.
- 사장님 케이스: 매출 4칸 0 + 현금상세 277,920 → 매출관리에 277,920 잘못 표시.

**검증**: 마감 기록 화면에서 `pos_cash + pos_cash_receipt` ≠ `현금상세 합` 이면 ⚠️ 직원 입력 누락 안내.

## 10. 영업개시 생략 가능 + 이월금 fallback (2026-05-15 신설)

**규칙**: 영업개시 보고는 **선택사항**. 매일 보고할 필요 없음.

**이월금 연속성 우선순위**:
1. 마감(N-1) → **영업개시(N)** → 마감(N) (영업개시 있을 때 — 도난·실수 검증 활성)

## 11. 카테고리별 매출 대비 경고 기준 % (threshold) — 2026-05-22 명문화

**기본값** (사장님이 카테고리 관리에서 매장별 수정 가능, `store_settings.expense_thresholds` JSONB):

| 카테고리 | 기본 기준선(%) | 비고 |
|---|---|---|
| 식자재 | 30 | 사장님 매장 5월 35.4% (기준 초과 ⚠️) |
| 주류 | 10 | |
| 음료 | 10 | |
| 인건비 | 25 | 시급+월급+상여금 자식 합산 |
| 비품 | 5 | |
| 마케팅 | 10 | |
| 고정비 | 15 | |
| 공과금 | 15 | |
| 세금 | 10 | |
| 기타 | 10 | |
| 로열티/수수료 | 0 | 매출×요율 자동 계산 |

**동작**:
- 매출 대비 비율 = 카테고리 금액 ÷ 매출 × 100 (실시간 계산, 고정 X)

**설정 위치**: 사이드메뉴 → 지출 카테고리 → 카테고리 편집 → 매출 대비 경고 기준(%) 입력란

## 12. 카드 매출 = POS 원액 / 계좌 입금액 ≠ 매출 (2026-05-24 사장님 명시)

> ⚠️ **사장님 도메인 지식 — 매출 산정 핵심**

### 규칙
- **카드 매출**은 반드시 **POS 원액**(`pos_card` / `sales_daily.card`) 기준
- 카드사가 매장 계좌로 입금하는 금액 = **카드 수수료 차감된 순액**이므로 매출 산정에 사용 절대 X
- 은행 입금 내역(`mydata_transactions.tx_type='bank'` 입금)에서 "카드사 입금"은 **매출 분류 절대 금지**

### 왜
- 카드 수수료율 = 가맹점·카드사·결제수단별로 다름 (보통 1.5~3%)
- 계좌 입금액 기준 매출 = 매출이 실제보다 1.5~3% 낮게 잡힘
- 매출 정확성 = vision PoC 완벽 기준 100% 일치의 핵심

### 그림(blueprint_diagram.svg) 반영 사항
- 은행카드(`mydata`) → 매출 분류 폴더 화살표 = **그리지 않음**
- 매출 단일 진실의 원천 = `sales_daily` (POS 원액 기반)
- 카드 수수료는 **별도 카테고리** (`card_fee_rate` × 카드매출, 정산검수에서 자동)

### CTO 페널티 (헌법 1-7 자가 박음)
- 2026-05-24 그림 v2 검토 시 CTO가 "은행카드 → 매출도 받음" 추측 박음 = 사장님 도메인 무시
- 사장님 정정: "수수료 뗀 거라 참고하면 안 됨"
- → 헌법 3-2 분업 위반 (식당 도메인은 사장님 압도, 추측 X)
