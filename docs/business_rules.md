### 0-8. 부가세(VAT) 처리 규칙 (영수증 — 2026-06-22 사장님 결정)

> 사장님 방향: "과세는 부가세 나오게, 면세는 아닌 거로." + "쿠팡도 부가세 알아서 잡고, 애매한 건 표시."
> **회계 정답**: 과세 매입의 부가세는 비용 아님 = 나중에 돌려받는 매입세액. 진짜 원가 = 공급가(세전). 단, 실제 신고는 세무사가 쿠팡 세금계산서 등으로 처리 (앱은 사장님 내부 가시성용).

**🚨 면세/과세 = 부가가치세법 기준만 따름 (임의 기준 금지 — 2026-06-23 사장님 "법을 따라야 변동성이 없다")**
> 근거 법령: 부가가치세법 시행령 제34조 + 시행규칙 제24조 별표1. 우리가 품목별로 임의 판단하면 자꾸 변동성 생김 → **법 하나만 단일 진실.**
> 출처: 국가법령정보센터 law.go.kr, 농식품부 atfis.or.kr 안내(별표1 개정 '22.7.1).

- **면세(true)** =
  ① 미가공 농·축·수·임산물 (생채소·생과일·정육·생선·쌀·곡물)
  ② **1차 가공** — 탈곡·정미·제분·정육·건조·냉동·염장·**데침** 등 *원생산물 본래 성질이 변하지 않는 정도* (예: 데친연근=데친채소=면세)
  ③ **단순가공식료품 11종** (시행규칙 별표1): 데친채소류·김치·단무지·장아찌·젓갈류·게장·두부·메주·간장·된장·고추장
  ④ 도서·신문
- **과세(false)** = 본질 가공식품(소시지·어묵·맛살·햄·과자·라면·소스·드레싱·통조림 등)·공산품·비품·주방용품·생활용품·주류·음료·생수·커피
- "어디서 사느냐"(쿠팡·거래처)가 아니라 "무엇이냐(법상 품목 성격)"로 정함.
- ⚠️ **2026.1.1 개정 주의**: 두부·메주·간장·된장·고추장은 "제조시설 갖춰 판매목적 독립거래단위로 포장·병입"하면 과세. 단순 운반·보관 포장은 면세 유지 → 식당 매입은 대부분 면세지만 **경계 품목은 세무사 확인**.
- 코드: `assets/common.js` 프롬프트 `f` 필드(거래처·온라인·마트 3채널 동일 법기준). 영수증에 세액(t) 찍혔거나 과세표시면 무조건 과세.

**채널별 부가세 산정 (receipt.js):**
| 채널 | 부가세(t) 출처 |
|---|---|
| 세액 칸/세액 총액 **있는** 영수증 | 영수증에 찍힌 **세액 총액(total_tax)을 진실로**, 품목별 부가세 합을 그 값에 비례 보정 (`_rcpReconcileTaxToPrinted`). AI가 면세 품목에 과세 잘못 붙여도 합이 영수증과 일치. 금액(낸 돈)·순익 불변 |
| 세액 칸 **없는** 과세 매입(쿠팡·순창국제 등 — 채널 무관, 주류 제외) | 앱이 과세분만 역산: `tax=round(p/11)`, 면세는 0 (`_rcpDeriveVatNoTaxLine`). 영수증에 세액 하나라도 찍혀 있으면 호출 안 함 |
| 수동 입력 | 사장님이 "부가세 포함" 토글 → `금액÷11` 자동, 면세는 안 누름 |

> 근거(사장님 2026-06-22): "찍힌 금액만큼만 이체한다 = 그 금액이 낸 돈 = 부가세 포함." → ÷11. ⚠️ 거래처별 부가세 포함/별도 방식은 **사장님 홈택스 확인 대기**(2026-06-23 예정). 별도 방식 거래처가 확인되면 거래처별 설정으로 분기 예정.

**표시 규칙 (buildReceiptRow 카드):**
- 단가 = **세전 단가**(공급가÷수량). 과세는 "공급가 + 부가세 = 합계" 회색 줄 표시. 면세는 단가×수량=금액(부가세 줄 없음).
- "부가세 포함" 토글 = **모든 카드**(사진·수동) 노출 → AI가 틀려도 사장님이 그 자리서 수정.
- AI가 면세/과세 애매하다고 본 식품(온라인 `vu=true`: 냉동·반가공·손질·양념·세트 등) = 카드에 **"💧 부가세 확인?"** 뱃지.
- DB: `receipts.tax_amount`(부가세)·`is_tax_free`(면세 여부) 저장됨. 금액(total_price=실제 낸 돈)·순익은 부가세와 무관하게 불변.

> ⚠️ 단가의 unit_price는 회계/순익 계산에 안 쓰임(매칭·검산·표시용). 과세 행만 세전 단가로 저장. 순익은 항상 total_price(세후) 기준.

## 8. 영업개시 보고 + 차감 추적 (2026-05-12)

- 두 화면 모두 동적 행 (type 셀렉트 + 금액 + 메모 + 행 추가/삭제)

### 차액 의미
| 차액 | 정상 (0) | 불일치 | 의미 |
|---|---|---|---|

## 9. 매출 = 마감 4칸 (2026-05-12 확립)

**규칙**: 마감 매출은 `현금 + 현금영수증 + 신용카드 + 기타결제` 4칸 합만으로 계산.

| 카테고리 | 기본 기준선(%) | 비고 |
|---|---|---|
| 식자재 | 30 | 사장님 매장 5월 35.4% (기준 초과 ⚠️) |
| 주류 | 10 | |
| 음료 | 10 | |
| 인건비 | 25 | 시급+월급+상여금 자식 합산 |
| 비품 | 5 | |
| 마케팅 | 10 | |
| 고정비 | 15 | |
| 공과금 | 15 | |
| 세금 | 10 | |
| 기타 | 10 | |
| 로열티/수수료 | 0 | 매출×요율 자동 계산 |

**동작**:
- 매출 대비 비율 = 카테고리 금액 ÷ 매출 × 100 (실시간 계산, 고정 X)
- 비율 > threshold → 정산현황 v17에서 ⚠️ 빨강 표시 (over 클래스)
- 비율 < threshold × 0.85 → 초록 (good 클래스) — 캘린더 셀 카테고리 모드
- threshold = 0 → 경고 없음

**설정 위치**: 사이드메뉴 → 지출 카테고리 → 카테고리 편집 → 매출 대비 경고 기준(%) 입력란

**저장 위치**: `store_settings.expense_thresholds = {"식자재": 30, "인건비": 25, ...}` JSONB

## 12. 카드 매출 = POS 원액 / 계좌 입금액 ≠ 매출 (2026-05-24 사장님 명시)

> ⚠️ **사장님 도메인 지식 — 매출 산정 핵심**

### 규칙
- **카드 매출**은 반드시 **POS 원액**(`pos_card` / `sales_daily.card`) 기준
- 카드사가 매장 계좌로 입금하는 금액 = **카드 수수료 차감된 순액**이므로 매출 산정에 사용 절대 X
- 은행 입금 내역(`mydata_transactions.tx_type='bank'` 입금)에서 "카드사 입금"은 **매출 분류 절대 금지**

### 왜
- 카드 수수료율 = 가맹점·카드사·결제수단별로 다름 (보통 1.5~3%)
- 계좌 입금액 기준 매출 = 매출이 실제보다 1.5~3% 낮게 잡힘
- 매출 정확성 = vision PoC 완벽 기준 100% 일치의 핵심

### 그림(blueprint_diagram.svg) 반영 사항
- 은행카드(`mydata`) → 매출 분류 폴더 화살표 = **그리지 않음**
- 매출 단일 진실의 원천 = `sales_daily` (POS 원액 기반)
- 카드 수수료는 **별도 카테고리** (`card_fee_rate` × 카드매출, 정산검수에서 자동)

### CTO 페널티 (헌법 1-7 자가 박음)
- 2026-05-24 그림 v2 검토 시 CTO가 "은행카드 → 매출도 받음" 추측 박음 = 사장님 도메인 무시
- 사장님 정정: "수수료 뗀 거라 참고하면 안 됨"
- → 헌법 3-2 분업 위반 (식당 도메인은 사장님 압도, 추측 X)

## 금고 이월 사슬 규칙 (2026-06-01 사장님 정의)

마감정산의 **시작 금고**(전일 이월금 칸)를 정하는 순서:
1. 그 날 **영업개시 실시함** → **영업개시 금고**

## 카테고리 계층 — 주류·음료는 식자재 하위 (2026-06-25 사장님 결정)

- **주류·음료 = 식자재의 소분류(자식)**. 육류·야채·공산품과 같은 계층.
  - 도입 배경: "주류·음료·식자재가 동일계층인데, 주류·음료를 식자재 하위로 넣자"(사장님). 식자재 합계에 주류·음료가 자동 포함되어야 회계가 맞음.
- **구현 방식 = DB 데이터만** (`expense_categories.parent_id`를 식자재 id로). 코드 0줄 수정.
  - 집계(`calcExpenseByCategories` 외)가 전부 parent_id 기반 자동 롤업 → 자식 금액이 부모(식자재)에 자동 합산.
  - 카테고리 `name`은 "주류"/"음료" 그대로 유지 → 아이콘·배지색·임계값 맵(이름 기반)은 안 깨짐.
- **순서**: 카테고리 관리 화면에서 ☰ 끌어 변경(sort_order, 자동 저장). 하드코딩 아님.
- 신규 매장 기본 시드엔 주류·음료 없음(매장이 직접 추가). 본 규칙은 추가했을 때 식자재 하위로 두라는 권고.
