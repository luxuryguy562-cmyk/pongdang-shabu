🔑 가장 중요한 발견

✅ 영수증(receipts)은 이미 품목별 칸이 있습니다

영수증은 줄마다 카테고리 칸(category_id)이 있습니다. 지금은 거래처 모드일 때만 이걸 무시하고 거래처 카테고리로 덮어씁니다(receipt.js:1485). 그 덮어쓰는 코드 2줄만 풀면 끝 — 사장님 말씀대로입니다.

⚠️ 거래처 주문내역(vendor_orders)은 품목별 칸이 아예 없습니다

거래처 주문은 거래처 단위 카테고리(vendors.category_id)로만 집계됩니다. 줄마다 카테고리를 넣으려면 새 칸을 만들고, 집계 7곳을 동시에 바꿔야 합니다. 한 곳만 빠뜨려도 합계가 틀어집니다. 이게 4단계가 위험한 이유입니다.

1"직구" → "마트·시장" 이름 변경 낮음

무엇: 화면에 보이는 "직구" 글자만 "마트·시장"으로. 동작은 100% 그대로.

건드릴 곳 (8곳):

  • 화면 글자 index.html: 모드 선택 버튼, 카테고리 카드, 목록 헤더
  • 배지·제목 receipt.js:148,346 / 토스트 안내

DB 변경: 없음. 충돌 방지: 글자만 바꿔서 가장 안전 — 이걸 먼저 합쳐서 본가 동기화 흐름 점검.

2거래처 카테고리 고정 풀기 (영수증) 중간

무엇: 거래처 영수증도 품목별로 카테고리를 정하게. 웰스토리에서 육류 사면 육류로 잡히게.

핵심 — 딱 2줄

receipt.js:1485, 1490 의 "거래처 모드면 거래처 카테고리로 강제" 로직을 "항상 줄별 선택값 사용"으로. 품목별 분류 화면(picker)은 이미 만들어져 있어서 켜기만 하면 됩니다.

✅ 규격·원산지·거래처 연결은 100% 유지 (사장님 지적)

거래처 영수증의 규격(spec)·원산지(origin) 분리 표시는 카테고리와 완전히 다른 코드입니다 (receipt.js:1510~1517 vs 카테고리 강제 1493~1496). 카테고리 덮어쓰는 2줄만 풀고, 규격·원산지·거래처 연결(vendor_id)은 손도 안 댑니다. 거래처 영수증은 지금처럼 규격·원산지가 따로 나옵니다.

⚠️ 주의 — 이건 영수증(receipts)만

거래처 주문내역(vendor_orders)은 4단계에서 따로. 기존 거래처 카테고리 데이터(vendors.category_id)는 지우지 않고 남깁니다 — 지금 집계가 안 깨지게(사장님: "바꿀 일 없다").

DB 변경: 없음.

3🌐온라인 신설 + 쿠팡 스샷분석 정리 중상

무엇: 영수증 종류에 "온라인" 추가(쿠팡·네이버·옥션 등). 거래처→쿠팡에 있던 스샷 분석은 제거(중복).

① 온라인 모드 추가 (13곳): 대부분 기존 "마트·시장(직구)" 흐름을 그대로 재사용. 차이는 상호=플랫폼 이름, 실제 판매자는 품목명에, 배송비 별도 — 이미 AI 규칙으로 넣어둔 부분 활용.

② 기존 쿠팡 스샷분석 제거 (함수 6개, 81줄 + 화면 2개):

✅ 살리는 것 / 죽이는 것 (사장님 지시대로)
  • 죽임: 거래처→쿠팡 "스샷 AI 분석" 버튼·시트 analyzeCoupangShots 등
  • 살림: 북마클릿 자동 동기화, 분류 대기함, 학습 규칙 — 나중에 쓰실 기능 그대로
⚠️ 발견된 한계 — 쿠팡 이름이 코드에 박혀있음

지금 vendor.name === '쿠팡'으로 3곳에 하드코딩(코드에 직접 박음)돼서, 네이버·옥션은 자동 동기화가 안 됩니다. 온라인을 제대로 하려면 거래처에 "온라인 종류" 표시 칸을 두는 게 맞습니다. → 이게 아래 선택 DB 변경입니다.

🗄️ DB 변경 (선택) — 거래처에 "온라인 여부" 칸

vendors 표에 integration_type(연동 종류) 칸 추가. 쿠팡·네이버·옥션을 이름 하드코딩 없이 구분. 기존 데이터 영향 없음(새 칸, 기본값 비움).

실행 SQL / 롤백 SQL은 승인 시 별도 제시 (헌법 8조).

4주문내역 품목별 카테고리 (집계) 높음

무엇: 거래처 주문내역도 줄별로 카테고리. 가장 큰 공사 — 맨 나중에 충분히 준비해서.

🗄️ DB 변경 (필수) — 주문내역에 카테고리 칸 추가

vendor_orders 표에 category_id 칸 신설. 기존 주문은 거래처 카테고리를 복사해 채움(데이터 안 깨짐). 이후 새 주문부터 품목별.

⚠️ 집계 7곳을 동시에 바꿔야 함 (한 곳만 빠져도 합계 틀어짐)
집계 함수위치영향
지출 카테고리 합계sidemenu calcExpenseByCategories ×3대시보드·정산 전체
홈 일별 카드 분배dashboard ×2홈 화면·월 요약
지출 허브 카드sidemenu loadExpHubData ×2지출관리 입구

전부 vendors.category_id → 주문.category_id로 한 번에 전환 + 검증 필수.

🛡️ 충돌·사고 방지 (전 단계 공통)

  • 각 단계 = 따로 합치기(PR) 1개 — 한 단계 끝낼 때마다 본가에 합치고 사장님이 앱에서 확인
  • 단계 시작 전 본가 동기화 — 코드 겹침 충돌 예방 (교훈 #137)
  • 단계마다 백업 커밋 + 문법 검사 + 잔재 검색 (헌법 11조)
  • DB 변경은 실행 SQL + 롤백 SQL 같이, 사장님 "실행 승인" 4글자 후만 (헌법 8조)
  • 기존 데이터 보존 — 거래처 카테고리 안 지움, 주문내역 칸은 복사로 채움
