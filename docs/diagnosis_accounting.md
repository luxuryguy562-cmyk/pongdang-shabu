## 1. 사장님 운영 모델 (확정 2026-06-21)

한 달 순익 = **매출 − 지출**

| 구분 | 항목 | 계산 |
|---|---|---|
| 매출 | 마감정산 입력 숫자 | `sales_daily` |
| 지출 | 카드수수료 | 카드매출 × 설정% |
| 지출 | 로열티 | 총매출 × 설정% |
| 지출 | 인건비(시급) | 근태 시간×시급 + 주휴수당 + 추가수당(주말·야간·특별) |
| 지출 | 인건비(월급) | 월급직원 월급 일할 |
| 지출 | 영수증·명세서 | 영수증/명세서 AI분석 또는 수기입력 |
| 지출 | 고정비·공과금 | **예상금액**일 땐 일할(진행일수), **실제금액** 넣으면 **일괄(월 전체)로 실제금액** |

## 4. 발견된 문제 (정리 대상)

### 4-1. 순익·지출 계산이 여러 곳에 중복 (헌법 7-7 위반) 🔴
- 같은 "한 달 순익"을 **여러 함수가 따로 계산**한다.
  - 홈 도넛 카드 (일별 분배 합산 방식) — 사장님이 보는 것, 영수증 전부 계상
  - 옛 월요약 "표" 계산 (`calcExpenseByCategories`) — **화면엔 안 뜨지만**(아래 4-2) 코드는 돎, 거래처 소스 영수증 누락
  - `supabase/functions/my-stores` (내 매장들 서버) — 처음엔 옛 표 방식 따라가 음료·주류 영수증 누락 → **2026-06-21 수정 완료**(영수증 전부 계상)
- 결과가 서로 달라질 수 있음. 실제로 **음료·주류 영수증 1,756,480원 사고**가 이 중복 때문에 발생.

### 4-2. 죽은 코드 (계산은 되는데 화면에 안 뜸) 🔴
- 옛 월요약 "표"(`summHtml`)를 그리는 코드가 남아있으나, 띄울 자리(`dashSummaryGrid`)가 화면 개편 때 삭제됨 → **계산만 하고 안 보임**. (근거: `dashboard.js` 858줄 주석 "옛 dashSummaryGrid … DOM 없음", `index.html`에 해당 DOM 없음)
- 옛 일별정산표(`dashDailyTable`)도 자리 없는 잔재.

### 4-3. 마이데이터(통장·카드 끌어오기) — 사장님 안 씀 → 제거 대상 🟡
- 사장님 결정(2026-06-21): *"마이데이터 안 쓸 거야."*
- 통장·카드 자동 내역 기반 "실제 마감(진마감)" 경로 전체. 여러 화면(지출관리·정산대조 등)에 퍼져 있음 → 범위 파악 후 통째 제거 예정.
- 안심: 논산점은 마이데이터 0건이라 제거해도 **현재 순익 숫자 안 바뀜**.

### 4-4. 고정비 실제금액: 일할 vs 일괄 — **정상 (오진 정정 2026-06-21)** ✅
- 사장님 뜻(명확화): 실제금액 넣으면 **하루 단가를 실제금액÷그달일수로 새로 잡아, 1일치부터 전부 다시 계산**(누적 = 새 단가 × 진행일수).
- 예) 예상 90만 → 하루 3만 → 15일 45만. 실제 99만 입력 → 하루 3.3만 → 15일 **49.5만**.
- 현재 코드: `round(실제금액/그달일수) × 진행일수` = 정확히 위와 동일. (`dashboard.js` 650~665, 서버 `my-stores/index.ts`, 지출관리 `sidemenu.js` 3688~3694 모두 같음)
- ⚠️ 앞서 CTO·진단도구가 "실제금액이면 일괄(전액) 표시여야 하는데 일할로 나뉘어 틀림"이라고 했으나 **오진**. 사장님 의도 = 일할 단가 재계산이며 코드가 그대로 함. **고칠 것 없음.**

### 4-5. 가마감(영수증·매출 기준)/진마감(통장·카드 기준) 두 모드 🟡
- **가마감** = 영수증·매출·근태·고정비로 계산. 홈 화면 기본. (사장님이 쓰는 것)
- **진마감** = 통장·카드 실제 출금(마이데이터)으로 계산. **마이데이터 필수 의존.**
- 진마감은 인건비를 마이데이터 출금만으로 잡아 근태 시급·월급을 무시함(`sidemenu.js` 3645~3647). → 마이데이터 제거 시 진마감도 같이 제거되어 이 문제 자동 소멸.

### 4-6. 마이데이터(통장·카드) 퍼진 위치 — 제거 범위
- 지출관리 탭 계좌·카드 내역: `sidemenu.js` `loadBankTransactions` 3859, `loadCardTransactions` 3978
- 진마감 지출 집계: `sidemenu.js` 3514~3522 (`calcExpenseByCategories` final 분기)
- 진마감 로열티: `dashboard.js` 554~558
- 정산대조(recon): `sidemenu.js` 7474 부근
- 자동 분류·삭제: `sidemenu.js` 4019, 4030, 7370~7390

### 4-7. v17(홈 도넛) = 화면 표시 단일 진실 (좋은 점)
- 홈 화면 순익 표시는 v17 계열 하나만 사용 → **표시 중복 없음**(좋음). 옛 표 계산은 화면엔 안 뜸(죽음).
- 다만 옛 표 계산 함수(`calcExpenseByCategories`)는 지출관리·정산대조에서 아직 쓰여서 통째 삭제는 불가, **고정비 일할 부분만 통일 수정** 필요.

## 5. 정리 방향 (3단계, 회계 안전 최우선)

1. **죽은 코드 제거** — 화면에 안 뜨는 옛 표 계산 등. 숫자 영향 0.
   - ✅ 2026-06-21: 옛 월요약 "표"(summHtml→#dashSummaryGrid) 빌드 101줄 제거 완료(`dashboard.js`). 구문 검증·잔재 0건 확인. 홈 도넛 카드 표시는 그대로(살아있는 값 유지).
   - ⏳ 남음: momCat(전월대비) 다운스트림(now null-safe dead), dashDailyTable 가드, 고아 핸들러(toggleExpMoreCategories 등) — 3단계 통일 때 함께 정리.
2. **마이데이터/진마감 제거** — 사장님 안 씀. 퍼진 범위 목록 먼저 → 백업 후 제거.
3. **순익·지출 계산 하나로 통합** (헌법 7-7) — 홈·내 매장들·어디서나 같은 함수. 매 단계 골든값(논산 903만 등) 대조 검산.

> 각 단계 백업 커밋 + `node --check` + 골든값 대조 필수 (헌법 11조).
